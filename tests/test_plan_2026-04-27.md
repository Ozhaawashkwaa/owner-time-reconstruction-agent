## Procédure de test

### Phase 1: Pré-flight (à faire avant le code)

```bash
# Dans Windsurf / Claude Code

# 3. Setup .env

# Remplir les valeurs

# 4. Install requirements

```

### Phase 2: Test download .db

```bash
# Test isolé du module
python routine/download_aw_db.py --target-date 2026-04-27 --output /tmp/aw_test.db
```

**Critères de succès:**
- Fichier .db téléchargé dans `/tmp/aw_test.db`
- Taille ~57 MB
- `file /tmp/aw_test.db` retourne "SQLite 3.x database"

**Si échec:**
- Si erreur permission Drive → vérifier OAuth MCP
- Si timeout → augmenter timeout, vérifier réseau
- Si fichier corrompu → réessayer

### Phase 3: Test parse SQL

```bash
# Test isolé du parser
python routine/parse_aw.py --db /tmp/aw_test.db --target-date 2026-04-27 --output /tmp/aw_events.json
```

**Critères de succès:**
- JSON file généré avec liste d'events du 27 avril
- Au minimum: bucket `aw-watcher-window` events parsés
- Champs: `timestamp`, `duration`, `app`, `title`, `bucket`
- Total durée active > 0 (Franck a forcément travaillé un peu)

**Vérifications manuelles:**
- L'event Daily Scrum 6:30-7:00 doit apparaître (Google Meet ou Chrome)
- Au moins quelques events de coding/email/etc.

### Phase 4: Test correlation

```bash
python routine/correlate_signals.py --events /tmp/aw_events.json --target-date 2026-04-27 --output /tmp/blocks.json
```

**Critères de succès:**
- Blocs de temps groupés (>15 min)
- Chaque bloc enrichi avec contexte (calendar event, emails, repo, etc.)
- Confidence score sur chaque bloc
- Daily Scrum identifié comme correspondant à event Calendar

### Phase 5: Test ClickUp mapping

```bash
python routine/map_to_clickup.py --blocks /tmp/blocks.json --output /tmp/proposed_entries.json
```

**Critères de succès:**
- Chaque bloc significatif a une `proposed_entry`
- Daily Scrum mappé vers MAIA > Solution Design (ou nouvelle tâche si pas trouvé)
- Email AUpoint mappé vers AUpoint space
- Fit quality scores cohérents

## Validation par Franck

Une fois le message dans Slack, Franck:

1. Lit l'analyse
2. Répond aux questions (off-computer, ambiguïtés)
3. Pour ce premier test, on **n'enclenche pas encore** le cycle complet conversation+APPROVE
4. Franck reporte:
   - ✅ Ce qui est correct dans l'analyse
   - ❌ Ce qui est faux
   - ⚠️ Ce qui manque

Cette validation va servir à **améliorer** la logique avant le test sur 28, 29, 30 avril.

## Métriques de succès du MVP

Pour considérer Phase 1 (test rétroactif) comme réussie:

- [ ] **Pipeline technique fonctionnelle:** Tous les modules tournent sans erreur
- [ ] **Précision >70%:** Au moins 70% des blocs de temps sont correctement mappés (selon validation Franck)
- [ ] **Off-computer demandé:** L'agent a bien posé la question
- [ ] **Réconciliation présente:** Le message inclut la math des totaux
- [ ] **Message lisible:** Franck peut comprendre l'analyse sans avoir à demander des clarifications sur le format
- [ ] **Aucune fausse création ClickUp:** Aucune entry créée sans APPROVE
- [ ] **Temps total <5 min:** L'analyse complète tourne en moins de 5 minutes

## Anti-tests (ce qu'on NE teste PAS encore)

- ❌ Cycle complet de conversation (Phase 2)
- ❌ Création réelle dans ClickUp (Phase 2)
- ❌ Update Google Sheet state (Phase 2)
- ❌ Apprentissage du mapping (Phase 2)
- ❌ Cron scheduler (Phase 4)
- ❌ Multi-jours en parallèle (jamais — toujours un jour à la fois)

---

## Si le test échoue lamentablement

Plan B:

1. Si le .db ne peut pas être téléchargé → debug Drive auth dans Claude Code
2. Si le parsing SQLite échoue → étudier le schéma de la DB ActivityWatch (tables `bucketmodel`, `eventmodel`)
3. Si la correlation ne donne rien → simplifier l'analyse (juste AW + Calendar pour commencer)
4. Si le mapping est complètement à côté → refaire le `clickup_workspace_mapping.md` avec Franck

**À retenir:** Mieux vaut un MVP qui fait moins mais bien, que tout faire mal.
